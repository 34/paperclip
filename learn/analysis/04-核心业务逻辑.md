# Paperclip 核心业务逻辑深度分析

## 1. 心跳系统 (Heartbeat) — 系统核心引擎

心跳系统是 Paperclip 最核心的子系统，负责 Agent 的执行调度和会话管理。代码量 159k，是最大的服务文件。

### 1.1 心跳执行流程

```
触发源                          心跳服务                         执行环境
──────┐                        ┌──────                         ┌──────
       │                        │                              │
定时器 │──timer──→               │                              │
       │                        │  1. 检查并发限制              │
任务   │──assignment──→          │  2. 获取启动锁               │
指派   │                        │  3. 构建上下文快照            │
       │                        │  4. 解析工作区                │
手动   │──on_demand──→           │  5. 解析/恢复会话             │
触发   │                        │  6. 选择适配器                │
       │                        │  7. 执行适配器                │──→ Agent 运行
自动化 │──automation──→          │  8. 收集日志/使用量           │
       │                        │  9. 记录成本                  │
       │                        │  10. 处理结果                 │
       │                        │  11. 更新 Issue 状态          │
       ▼                        ▼                              ▼
```

### 1.2 并发控制

```typescript
// 每个 Agent 的最大并发运行数（默认 1，可配置到 10）
maxConcurrentRuns: runtimeConfig.maxConcurrentRuns || 1

// 启动锁：per-agent Promise 链防止竞态
const startLocksByAgent = new Map<string, Promise<void>>()
```

### 1.3 会话管理

**会话类型适配器**：`claude_local`, `codex_local`, `cursor`, `gemini_local`, `opencode_local`, `pi_local`

**会话键 (Task Key) 派生**：
```
contextSnapshot.taskKey → contextSnapshot.taskId → contextSnapshot.issueId
→ payload.taskKey → payload.taskId → payload.issueId
→ "__heartbeat__" (回退键，用于定时唤醒)
```

**会话生命周期**：
```
新会话创建 → 会话使用（跨心跳保持） → 会话压缩（策略驱动） → 会话轮换
                 │
                 ├── 工作区匹配检查：前次会话 CWD 是否匹配当前工作区
                 ├── 强制新会话条件：forceFreshSession 或 wakeReason=issue_assigned
                 └── 会话迁移：项目工作区可用时自动迁移
```

### 1.4 唤醒上下文构建

```json
{
  "wakeReason": "comment_added|issue_assigned|timer|...",
  "issueId": "uuid",
  "commentId": "uuid",
  "wakeCommentIds": ["uuid1", "uuid2"],
  "wakeSource": "agent|board|system",
  "wakeTriggerDetail": "manual|ping|callback|system",
  "taskKey": "derived-task-key"
}
```

**唤醒评论批处理**：
- 最多 8 条内联评论
- 每条评论正文最多 4000 字符
- 总计最多 12000 字符
- 构建 `paperclipWake` 负载：Issue 摘要 + 评论正文

### 1.5 工作区解析策略

```
解析顺序:
  1. project_primary    → 项目主工作区
  2. task_session       → 任务会话关联的工作区
  3. agent_home         → Agent 默认工作目录

工作区策略:
  - git_worktree: 自动 clone git 仓库（10分钟超时）
  - project_primary: 使用项目主目录
```

### 1.6 日志管理

```
日志存储策略:
  ├── 内联摘要: stdoutExcerpt/stderrExcerpt (MAX_EXCERPT_BYTES 上限)
  └── 外部存储: logStore + logRef + logSha256 + logCompressed

进程存活检测:
  ├── process.kill(pid, 0) 检测
  ├── EPERM → 存活
  └── ESRCH → 已死亡
```

## 2. Issue 生命周期系统

### 2.1 状态机

```
                    ┌──────────┐
                    │ backlog  │ ← 初始状态
                    └────┬─────┘
                         │
                    ┌────▼─────┐
              ┌─────│   todo   │
              │     └────┬─────┘
              │          │
              │     ┌────▼──────┐     ┌────────────┐
              │     │in_progress│◄────│  blocked   │
              │     └────┬──────┘     └──────┬─────┘
              │          │                    │
              │     ┌────▼──────┐             │
              │     │ in_review │─────────────┘
              │     └────┬──────┘
              │          │
              │     ┌────▼─────┐
              └────►│   done   │
                    └──────────┘
              ┌─────► cancelled
              │
              └── 任意状态 → cancelled
```

### 2.2 原子分配 (Checkout)

```
Issue.checkoutRunId 是并发控制锁:

Agent A 尝试 checkout Issue X:
  1. 检查 checkoutRunId 是否为 null
  2. 如果 null → 设置为当前 heartbeatRun.id → 分配成功
  3. 如果非 null → 检查对应运行是否已终止
     ├── 已终止 → adoptStaleCheckoutRun() → 过继锁
     └── 运行中 → 分配失败

Agent B 尝试 checkout 同一 Issue:
  → 发现 checkoutRunId 已被 Agent A 占用
  → 分配失败
```

### 2.3 阻塞关系

```
Issue A ──blocks──→ Issue B

规则:
  - Issue 不能阻塞自身
  - 检测循环阻塞链
  - 使用 issueRelations 表，type="blocks"
```

### 2.4 Inbox 机制

```
未读判断逻辑:
  最后外部评论时间 vs 用户最后触时间 (最后阅读/最后评论/创建时间/指派时间)

状态管理:
  - IssueReadState: 记录用户已读状态
  - IssueInboxArchive: 收件箱归档
```

## 3. 例行任务系统 (Routine)

### 3.1 执行流程

```
触发器触发 (schedule/webhook/api/manual)
     │
     ▼
创建 RoutineRun (status: received)
     │
     ▼
检查并发策略 ──────────────────────────────┐
     │                                     │
     ├── coalesce_if_active                │
     │   └── 有活跃 Issue → 合并上下文      │
     │       → status: coalesced            │
     │                                      │
     ├── skip_if_active                     │
     │   └── 有活跃 Issue → 跳过            │
     │       → status: skipped              │
     │                                      │
     └── always_enqueue                     │
         └── 总是创建新 Issue               │
            │                               │
            ▼                               │
     创建 Issue (originKind: routine_execution) │
     → status: issue_created                │
     → 触发 Agent 心跳执行                   │
                                            │
     补执行策略 ◄────────────────────────────┘
     ├── skip_missed: 跳过错过的执行
     └── enqueue_missed_with_cap: 补执行（上限25次）
```

### 3.2 Cron 调度

```
时区感知的 Cron 解析:
  - 使用 Intl.DateTimeFormat 处理时区
  - 逐分钟迭代计算下次执行时间
  - 最大迭代: 366 * 24 * 60 * 5 步
  - 存储在 routineTriggers.nextRunAt
```

### 3.3 变量系统

```typescript
// 支持的变量类型
type VariableType = 'text' | 'textarea' | 'number' | 'boolean' | 'select'

// 规则:
// - schedule 触发器: 所有必填变量必须有默认值
// - webhook 触发器: 负载可提供变量值
```

## 4. 预算控制系统

### 4.1 三级预算架构

```
┌─────────────────────────────────────────────────┐
│                  公司级预算                       │
│  ┌───────────────────────────────────────────┐  │
│  │              项目级预算                    │  │
│  │  ┌───────────────────────────────────┐    │  │
│  │  │           Agent 级预算            │    │  │
│  │  │  warnPercent: 80% → 警告          │    │  │
│  │  │  hardStop: 100% → 自动暂停        │    │  │
│  │  └───────────────────────────────────┘    │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 4.2 预算状态计算

```
观察金额 (从 CostEvent 计算):
  ok       → 金额 < 警告阈值
  warning  → 警告阈值 <= 金额 < 硬上限
  hard_stop → 金额 >= 硬上限

执行动作:
  ok       → 正常运行
  warning  → 发送通知（如果 notifyEnabled）
  hard_stop → 暂停实体（如果 hardStopEnabled）
              pauseReason = "budget"
              可能创建审批请求
```

### 4.3 恢复规则

```
预算暂停后的恢复:
  → 只恢复 pauseReason === "budget" 的暂停
  → 不覆盖手动暂停 (pauseReason === "manual")
  → 需要调整预算后才能恢复
```

```
两种模式:
  managed  → Paperclip 控制文件树
             路径: ~/.paperclip/companies/{companyId}/agents/{agentId}/instructions/
```

```typescript
  instructionsRootPath: '/path/to/instructions/',
```

## 6. 审批工作流

### 6.1 审批类型与触发

| 类型 | 触发条件 |
|------|----------|
| `hire_agent` | `requireBoardApprovalForNewAgents = true` 时创建新 Agent |
| `approve_ceo_strategy` | CEO Agent 提交战略方案 |
| `budget_override_required` | 预算超限需要覆盖 |
| `request_board_approval` | Agent 主动请求 Board 审批 |

### 6.2 审批状态机

```
pending ──► approved    (通过)
   │    ──► rejected    (拒绝)
   │    ──► revision_requested (要求修改)
   │    ──► cancelled   (取消)
   │
revision_requested ──► pending (重新提交)
```

## 8. 成本追踪系统

### 8.1 成本事件记录

```
每次 Agent 运行:
  1. 适配器报告 Token 使用量
  2. 计算增量 (当前 - 上次)
  3. 归一化计费类型:
     - metered_api: 按量计费
     - subscription_included: 订阅内（成本归零）
     - subscription_overage: 超量
     - credits: 积分
     - fixed: 固定费用
  4. 创建 CostEvent
  5. 检查预算策略
```

### 8.2 财务事件类型

14 种财务事件覆盖完整财务生命周期：
`inference_charge` | `platform_fee` | `credit_purchase` | `credit_usage` | `subscription_charge` | `subscription_credit` | `refund` | `adjustment` | `budget_allocation` | `budget_deallocation` | `budget_warning` | `budget_exceeded` | `invoice` | `payment`

## 核心业务流程总结

### 完整工作流示例：从创建任务到完成

```
1. Board 创建 Issue
   → Issue 状态: backlog
   → Activity Log 记录

2. Board 指派 Issue 给 Agent
   → Issue 状态: todo
   → Issue.assigneeAgentId 设置
   → 触发 assignment 心跳

3. Agent 执行心跳
   → Agent 状态: running
   → Issue.checkoutRunId 锁定
   → Issue 状态: in_progress
   → 创建/恢复会话
   → 解析工作区
   → 通过适配器执行

4. Agent 工作中
   → 定期报告进度 (Comment)
   → Token 使用量追踪
   → 成本事件记录
   → 预算检查

5. Agent 提交审查
   → Issue 状态: in_review
   → 执行策略检查签收
   → 如果需要 Board 签收 → 创建审批

6. Board 审查
   → 批准: Issue 状态 → done
   → 拒绝: 评论反馈 → Agent 继续工作
   → 要求修改: Issue 回到 in_progress

7. 完成
   → Issue 状态: done
   → completedAt 设置
   → 工作区标记为可清理
   → 会话归档
```
