## Architecture Overview

整个项目基于 Paperclip 虚拟公司架构运行。PPT代做服务不是一个独立应用，而是在 Paperclip 平台上创建一个虚拟公司，由 AI Agent 组成的团队自主运营。

## 1. Agent 设计

### 1.1 Agent 角色定义

| Agent | Role | reportsTo | 职责 | Adapter |
|-------|------|-----------|------|---------|
| CEO | `ceo` | (none) | 战略决策、定价调整、异常升级处理、数据报表审阅 | `claude_local` |
| 客服专员 | `pm` | CEO | 需求理解、企微客户沟通、报价、大纲确认、交付跟进 | `claude_local` |
| PPT设计师 | `designer` | 客服专员 | 大纲生成、内容填充、PPT渲染、修订迭代 | `claude_local` |
| 质检员 | `qa` | CEO | 质量检查、模板适配验证、输出评分、退回不合格品 | `claude_local` |
| 营销专员 | `cmo` | CEO | 小红书/闲鱼内容生成、投放素材制作、渠道ROI分析 | `claude_local` |

### 1.2 Agent 协作流程

```
获客入口:
┌──────────────┐  ┌──────────────┐
│ 小红书 (内容)  │  │  闲鱼 (商品)  │
│ → 群聊→企微   │  │ → 平台内成交  │
└──────┬───────┘  └──────┬───────┘
       │                 │
       ▼                 ▼
[客服专员] ─── 通过企微客服API/闲鱼聊天 解析需求
    │
    ├── 需求模糊? → 人工介入 (CEO审批)
    │
    ▼
[客服专员] ─── 向客户确认需求理解 + 报价
    │
    ├── 客户确认 + 付款
    │
    ▼
[客服专员] ─── 创建 Issue (status: todo, assignee: PPT设计师)
    │
    ▼
[PPT设计师] ─── checkout Issue
    │            ├── Step 2: 生成大纲
    │            ├── 大纲 → 客服专员转客户确认
    │            ├── Step 3: 内容填充
    │            └── Step 4: 调用 pptgen-plugin 生成 .pptx
    │
    ▼
[质检员] ─── checkout Issue (quality gate)
    │           ├── 评分 ≥ 7/10 → 通过
    │           └── 评分 < 7/10 → changes_requested, 退回设计师
    │
    ▼
[客服专员] ─── 生成水印预览PDF → 通过企微/闲鱼发送客户
    │
    ├── 客户付款 (企微: 手动确认 / 闲鱼: 担保交易) → 发送无水印原件
    └── 客户要求修改 → 创建子Issue, 回到设计师
```

### 1.2b 营销流程 (营销专员独立工作流)

```
[营销专员] ─── 每周触发 Routine
    │
    ├── 读取本周成功订单 (Issue status=done)
    │   ├── 提取前后对比素材
    │   ├── 生成小红书笔记文案 + 封面图建议
    │   └── 生成闲鱼商品描述优化建议
    │
    ├── 生成内容素材 → 人工审核后发布
    │   ├── 小红书: 前后对比帖 / 痛点帖 / 干货帖
    │   └── 闲鱼: 商品标题/描述优化
    │
    └── 渠道ROI报告 → CEO审阅
        ├── 薯条/聚光投放效果
        ├── 闲鱼推广效果
        └── 建议预算调整
```

## 2. Issue 生命周期映射

### 2.1 Issue Status → 订单状态

| Issue Status | 订单阶段 | 触发条件 |
|-------------|---------|---------|
| `backlog` | 待确认需求 | 企微客服/闲鱼消息进入系统，自动创建 |
| `todo` | 待制作 | 客户确认需求+报价，客服创建任务 |
| `in_progress` | 制作中 | 设计师 checkout |
| `in_review` | 质检中 | 设计师完成，质检员 checkout |
| `blocked` | 等待客户反馈 | 需要客户确认大纲或付款 |
| `done` | 已交付 | 客户确认收到文件 |
| `cancelled` | 已取消/退款 | 客户取消或退款 |

### 2.2 Issue 字段映射

```typescript
// 订单 Issue 模板
{
  title: "[订单] 毕业答辩PPT - {客户昵称}",
  description: "## 客户需求\n{原始消息}\n\n## 结构化需求\n{JSON}",
  status: "backlog",
  priority: "medium",  // 加急订单: "high", 2小时加急: "critical"
  assigneeAgentId: null,  // backlog状态不分配
  labels: ["ppt-order", "graduation-defense"],
  parentId: null,  // 修订订单的 parentId = 原始订单 Issue ID

  // 执行策略: 质检环节需要审批
  executionPolicy: {
    mode: "normal",
    commentRequired: true,
    stages: [
      {
        type: "review",
        approvalsNeeded: 1,
        participants: [{ type: "agent", agentId: "{质检员ID}" }]
      }
    ]
  }
}
```

### 2.3 Issue Documents (结构化数据)

每个订单 Issue 附加多个 Document:

| Document Key | 内容 | 创建者 | 更新时机 |
|-------------|------|--------|---------|
| `customer-requirements` | 结构化需求JSON | 客服专员 | 需求解析后 |
| `outline` | 逐页大纲 | PPT设计师 | 大纲生成后 |
| `content` | 逐页详细内容 | PPT设计师 | 内容填充后 |
| `quality-report` | 质检评分和反馈 | 质检员 | 质检完成后 |
| `revision-log` | 修改记录 | 客服专员 | 每次修订 |

### 2.4 Issue Attachments (文件)

| 文件类型 | 用途 | 创建者 |
|---------|------|--------|
| `output.pptx` | 最终PPT文件 | PPT设计师 |
| `preview.pdf` | 水印预览版 | 客服专员 |
| `outline.pdf` | 大纲预览 | PPT设计师 |

### 2.5 Issue Relations (依赖关系)

- 修订子订单 `blocks` 原始订单 (修订完成才能关闭原始订单)
- 加急订单之间互相 `blocks` (防止设计师同时处理多个加急)

## 3. Plugin 设计

### 3.1 wecom-plugin (企业微信客服 API 桥接)

**功能:** 通过企业微信客服 API 连接客户微信与 Paperclip Issue 系统

> **重要变更:** 原方案使用 Wechaty 桥接个人微信，调查发现 Wechaty 自2022年冻结，所有免费puppet协议已被封杀，主流Hook方案(Gewechat等)因法律风险被迫停。企业微信客服API是唯一合规、稳定、免费的路径。详见 `../research/微信桥接调查.md`。

**实现要点:**
- 注册企业微信 (免费)，开通微信客服功能
- 客户扫码即可开始对话，无需添加好友
- 支持发送11种消息类型：文本、图片、语音、视频、**文件**、链接、小程序卡片等
- Plugin Worker 接收 webhook 回调，解析消息后创建/更新 Issue
- **零封号风险** — 官方合规API
- 48小时回复窗口：客户最后一次交互后48小时内可主动发消息
- 通过 Paperclip Plugin State 存储客户映射关系:


## 4. Skill 设计

### 4.1 ppt-generation Skill (PPT设计师专用)

安装到 PPT设计师 Agent 上,定义生成工作流:

```markdown
# Skill: PPT Generation Workflow

## 触发条件
Issue 被分配给自己且 status = "todo", label 包含 "ppt-order"

## 工作流

### Step 1: 读取需求
- 读取 Issue 的 `customer-requirements` document
- 解析结构化JSON: ppt_type, topic, page_count, audience, tone, content_source, deadline

### Step 2: 选择模板
- 调用 `pptgen-plugin.list_templates` 获取匹配模板
- 根据 ppt_type 和 audience 选择最佳模板
- 在 Issue 评论中记录模板选择

### Step 3: 生成大纲
- 基于需求和模板 layout 选项生成逐页大纲
- 写入 Issue `outline` document
- 通知客服专员转客户确认 (创建子Issue, blocked by 客户确认)

### Step 4: 内容填充
- 客户确认大纲后继续
- 基于大纲和客户原始内容填充每页详细内容
- 写入 Issue `content` document

### Step 5: 渲染PPT
- 调用 `pptgen-plugin.generate_ppt`
- 首选: python-pptx 模板填充 (质量 8-9/10)
- 降级: 调用 `pptgen-plugin.call_chatppt_api` (API故障时)
- 将输出文件附加到 Issue
- 更新 Issue status 为 in_review

### 修订流程
- 收到 changes_requested 后
- 读取 `quality-report` document 中的修改意见
- 针对性修改,不重新生成全部
- 更新 `content` document 和附件
```

### 4.2 customer-service Skill (客服专员专用)

```markdown
# Skill: Customer Service Workflow

## 触发条件
- 企微客服新消息到达 (webhook 触发)
- 闲鱼新消息到达 (webhook/轮询触发)
- Issue status 变为 in_review (质检完成)
- Issue 变更需要客户通知

## 工作流

### 新消息处理 (双渠道)
1. 识别消息来源: 企微 or 闲鱼
2. 企微: 调用 `wecom-plugin.get_customer_info` 获取客户信息
3. 闲鱼: 从消息中提取买家信息
4. 查找 Plugin State 中是否有该客户的映射
5. 无映射: 创建新 Issue (backlog), 解析需求
6. 有映射且有活跃 Issue: 在该 Issue 上添加评论

### 需求解析
1. 分析客户消息文本
2. 提取: PPT类型, 主题, 页数, 截止时间, 风格偏好
3. 生成报价 (基于定价表, 渠道差异化定价)
   - 小红书来源客户: 300-2000元 (中高端)
   - 闲鱼来源客户: 30-100元 (走量)
4. 写入 `customer-requirements` document
5. 通过对应渠道向客户确认需求和报价

### 交付流程 (双渠道)
1. Issue 从 in_review 变为可交付
2. 调用 `pptgen-plugin.generate_watermark_pdf`
3. 企微渠道: 通过 `wecom-plugin.send_wecom_file` 发送预览
4. 闲鱼渠道: 通过网盘链接发送预览
5. 等待客户付款确认
   - 企微: 手动转账确认 或 聚合支付

### 修订协调

3. 创建子 Issue (assignee: PPT设计师, blockedBy: 原始Issue)
4. 跟踪修改次数 (Plugin State 记录)
```

```python
def generate_ppt(template_path: str, content: list[dict], output_path: str):
    """Fill a pre-designed .pptx template with structured content."""
    prs = Presentation(template_path)

    for page_content in content:
        layout_name = page_content["layout"]
        # 找到对应的 slide layout
        slide_layout = find_layout_by_name(prs.slide_layouts, layout_name)
        slide = prs.slides.add_slide(slide_layout)

        # 填充占位符
        for shape in slide.placeholders:
            if shape.placeholder_format.type == PL_TITLE:
                shape.text = page_content.get("title", "")
                apply_font(shape, font_name="思源黑体", size=28)
            elif shape.placeholder_format.type == PL_BODY:
                if "bullets" in page_content:
                    for i, bullet in enumerate(page_content["bullets"]):
                        if i == 0:
                            shape.text = bullet
                        else:
                            shape.text_frame.add_paragraph().text = bullet
                elif "body" in page_content:
                    shape.text = page_content["body"]
                apply_font(shape, font_name="思源黑体", size=18)

        # 处理图片提示(搜索Unsplash并插入)
        if "image_hint" in page_content:
            image_url = search_unsplash(page_content["image_hint"])
            image_path = download_image(image_url)
            add_image_to_slide(slide, image_path)

    prs.save(output_path)
```

## 7. 水印预览实现

```python
def generate_watermark_preview(pptx_path: str, output_pdf: str, watermark_text: str, clear_pages: int = 3):
    """
    生成水印预览版PDF:
    - 前3页: 清晰预览(吸引客户)
    - 第4页起: 加水印(保护内容)
    - 使用 LibreOffice headless 转换 PPTX → PDF
    - 使用 PyPDF 叠加水印层
    """
    # Step 1: PPTX → PDF (LibreOffice headless)
    subprocess.run([
        "libreoffice", "--headless", "--convert-to", "pdf",
        "--outdir", temp_dir, pptx_path
    ], check=True)

    # Step 2: 叠加水印
    reader = PdfReader(temp_pdf)
    writer = PdfWriter()

    for i, page in enumerate(reader.pages):
        if i >= clear_pages:
            # 创建水印页
            watermark_page = create_watermark_page(
                page.mediabox,
                text=watermark_text,
                opacity=0.3,
                rotation=45
            )
            page.merge_page(watermark_page)
        writer.add_page(page)

    writer.write(output_pdf)
```

## 9. 关键技术决策

### 9.1 PPT生成方案优先级

```
1. python-pptx + 自有模板填充 (质量8-9/10, 成本最低)
   ↓ 模板不匹配时降级
2. ChatPPT API / AiPPT API (质量7-8/10, 成本较高)
   ↓ API故障时降级
3. 人工介入 (CEO Agent创建Issue请求人工协助)
```

### 9.2 微信消息获取方案

```
首选: 企业微信客服 API (官方合规)
  - 零封号风险,免费,支持文件传输
  - 客户扫码即可对话,无需添加好友
  - 48小时回复窗口限制
  - 需要企业微信注册 (免费)

备选: wxauto (UI自动化, 仅用于获客初期)
  - 模拟真实用户操作,低封号风险
  - 需要专用Windows机器
  - 仅用于小红书/闲鱼引流后的第一个触点
  - 不适合规模化运营
```

详见调查报告: `../research/微信桥接调查.md`

### 9.3 支付方案

```
阶段1: 手动转账 + 客服确认 (最简单)
  - 客户微信转账
  - 客服专员标记付款状态 (Plugin State)

阶段2: 接入聚合支付 (正式运营后)
  - 使用第三方聚合支付(如 Ping++ / 收钱吧)
  - 自动确认到账,触发交付
```
