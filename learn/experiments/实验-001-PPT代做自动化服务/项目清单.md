## P1: PPT生成技术验证

**状态:** 未开始
**优先级:** 最高 (其他所有项目都依赖PPT能生成)
**预估时间:** 2-3天
**验证目标:** AI生成的PPT质量能达到"淘宝50元/页"水准

### 任务清单

- [ ] T1.1 搭建python-pptx开发环境
- [ ] T1.2 测试ChatPPT API输出质量 (联系必优科技获取API Key)
- [ ] T1.3 测试AiPPT API输出质量 (open.aippt.cn)
- [ ] T1.4 用同一份毕业答辩需求, 分别用3种方式生成PPT:
  - python-pptx + 简单模板填充
  - ChatPPT API
  - AiPPT API
- [ ] T1.5 质量盲评: 将3份输出 + 1份淘宝50元/页PPT给3个人盲评打分
- [ ] T1.6 确定首选技术方案和降级方案

### 输出物
- `learn/experiments/ppt-gen-validation/` — 测试代码和输出样本
- `learn/experiments/ppt-gen-validation/quality-report.md` — 质量对比报告

### 决策点
- 如果python-pptx+模板 ≥ 7/10 → 首选自建方案
- 如果ChatPPT API ≥ 8/10 且成本可控 → 阶段1用API
- 如果都 < 7/10 → 项目风险极大, 需重新评估

### 预算
- LLM API: ~10元
- ChatPPT API: 待确认 (可能需要企业合作)
- AiPPT API: 待确认

## P7: 上线运营验证

**状态:** 未开始
**优先级:** 最高 (项目的最终验证)
**前置:** P6 ✓, P3 ✓ (渠道就绪)
**预估时间:** 2周 (持续运营)
**验证目标:** 真实客户愿意付钱

### 任务清单

- [ ] T7.1 接第1个真实订单 (闲鱼, 门槛最低)
- [ ] T7.2 交付第1单, 收集客户反馈
- [ ] T7.3 接第2-5单, 优化交付流程
- [ ] T7.4 小红书引流订单开始进来 (如果薯条投流有效)
- [ ] T7.5 记录每单的: 渠道来源、咨询时间、成交金额、交付时间、客户评分
- [ ] T7.6 第4周末: 汇总数据, 决定是否继续投入

### 输出物
- `learn/experiments/operations/order-log.md` — 订单记录
- `learn/experiments/operations/retrospective.md` — 4周复盘

### 验证通过标准

| 指标 | 通过线 | 测量方式 |
|------|--------|---------|
| 有效咨询 | ≥ 25条 | Issue创建数 |
| 成交订单 | ≥ 10单 | Issue status=done |
| 总收入 | ≥ 2,000元 | 实际到账 |
| 平均交付时间 | < 4小时 | Issue时间差 |
| 客户好评率 | > 80% | 反馈记录 |
| 退款率 | < 10% | Issue cancelled |
| 投流ROI | > 5:1 | 收入/广告花费 |

### 决策点
- 全部达标 → 项目成立, 进入阶段2 (内部工具化)
- 部分达标 → 分析哪个环节是瓶颈, 针对性优化后再验证
- 严重不达标 (咨询<5条或0成交) → 需求不成立, 停止投入

## 关键风险

| 风险 | 影响 | 触发条件 | 应对 |
|------|------|---------|------|
| PPT质量不达标 | 项目不可行 | P1盲评 < 7/10 | 切换到ChatPPT API; 或放弃 |
| 无客户咨询 | 需求不成立 | P3第2周咨询=0 | 换内容策略; 换渠道 |
| 客户不付款 | 模式不成立 | P7成交率 < 5% | 调整定价; 先收钱再制作 |
| 闲鱼被封 | 失去低端渠道 | P3b账号被封 | 不依赖单一渠道; 转小红书 |
| Agent协作失败 | 无法自动化 | P6端到端跑不通 | 退回人工服务模式 |
