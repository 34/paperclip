# 人类用户与权限计划（Humans and Permissions Plan）

状态：草稿  
日期：2026-02-21  
负责人：Server + UI + Shared + DB

## 目标（Goal）

在一等支持人类用户与权限的同时，保留两种部署模式：

- 本地可信单用户模式，无登录摩擦
- 云端托管多用户模式，强制认证与授权

## 为何需要本计划（Why this plan）

当前 V1 假设围绕单一董事会运营者。现需要：多人类协作与按用户权限、安全的云端默认（避免无登录的生产）、本地模式仍可即开即用（`npx paperclipai run`）、agent 向人类的任务委派（含人类收件箱）、单账号多公司访问、实例管理员管理全实例公司访问、加入审批作为可操作收件箱提醒而非仅管理员页、人类与 agent 对称的邀请与审批入驻路径、人类与 agent 共享的成员与权限模型。

## 产品约束（Product constraints）

1. 所有新表、端点与权限检查严格保持公司作用域。
2. 保留现有控制平面不变式：单一负责人任务、审批闸门、预算硬停止、变更活动日志。
3. 本地模式保持简单可信，但避免不安全的云端配置。

## 部署模式（Deployment modes）

### 模式 A：`local_trusted`

行为：无登录 UI、浏览器直接进入 board 上下文、嵌入式 DB 与本地存储默认、存在本地隐式人类主体用于归属、该主体拥有有效 `instance_admin` 权限、邀请/审批/权限设置流程在本地模式仍可用（含 agent 注册）。护栏：默认仅绑定回环；若 `local_trusted` 且非回环绑定则启动失败；UI 显示持久「Local trusted mode」徽章。

### 模式 B：`cloud_hosted`

行为：所有人类端点需登录、Better Auth、初始为邮箱+密码、V1 不要求邮箱验证、支持托管 DB 与远程部署、多用户会话与角色/权限执行。护栏：缺少认证提供方/会话配置则启动失败；存在不安全绕过标志则启动失败；健康检查负载包含模式与认证就绪状态。

## 认证选择（Authentication choice）

使用 Better Auth 作为人类用户认证；V1 仅邮箱+密码登录；V1 不要求邮箱确认。（其余实现细节见英文原文。）
