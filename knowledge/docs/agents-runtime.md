 # Agent Runtime Guide（Agent 运行时指南）
 
 状态：面向用户的指南  
 最近更新：2026-02-17  
 目标读者：在 Paperclip 中配置和运行 agents 的运营者
 
 ## 1. 这个系统做什么
 
 在 Paperclip 中，agents 并不是持续运行的。  
 它们以 **心跳（heartbeat）** 的形式运行：由唤醒事件触发的一小段执行窗口。
 
 每一次心跳会：
 
 1. 启动为该 agent 配置好的适配器（例如 Claude CLI 或 Codex CLI）
 2. 向适配器提供当前的提示词/上下文
 3. 让它运行，直到正常退出、超时或被取消
 4. 存储结果（状态、token 使用量、错误、日志）
 5. 实时更新 UI
 
 ## 2. agent 何时被唤醒
 
 一个 agent 可以通过四种方式被唤醒：
 
 - `timer`：按计划的时间间隔（例如每 5 分钟）
 - `assignment`：当有工作被分配/领取到该 agent 时
 - `on_demand`：手动唤醒（按钮/API）
 - `automation`：用于未来自动化场景的系统触发唤醒
 
 如果某个 agent 已经在运行，新触发的唤醒事件会被合并（coalesced），而不是并发启动多个运行。
 
 ## 3. 每个 agent 需要配置什么
 
 ### 3.1 适配器选择（Adapter choice）
 
 常见选项：
 
 - `claude_local`：运行本机的 `claude` CLI
 - `codex_local`：运行本机的 `codex` CLI
 - `process`：通用的本地 shell 命令适配器
 - `http`：调用外部 HTTP 端点
 
 对于 `claude_local` 和 `codex_local`，Paperclip 默认认为 CLI 已经在主机中安装并完成认证。
 
 ### 3.2 运行时行为（Runtime behavior）
 
 在 agent 的运行时设置中，配置心跳策略：
 
 - `enabled`：是否允许定时心跳
 - `intervalSec`：定时器间隔（0 表示禁用）
 - `wakeOnAssignment`：在被分配工作时是否唤醒
 - `wakeOnOnDemand`：是否允许“按需唤醒”（ping 式）
 - `wakeOnAutomation`：是否允许系统自动化唤醒
 
 ### 3.3 工作目录与执行限制
 
 对于本地适配器，需要设置：
 
 - `cwd`（工作目录）
 - `timeoutSec`（每次心跳的最⻓运行时间）
 - `graceSec`（超时/取消后强制终止前的宽限时间）
 - 可选的环境变量与额外 CLI 参数
 - 使用 agent 配置中的 **Test environment** 来在保存前运行适配器相关的诊断
 
 ### 3.4 提示词模板（Prompt templates）
 
 你可以配置：
 
 - `promptTemplate`：用于每一次运行（包括首次运行与恢复会话）
 
 模板支持诸如 `{{agent.id}}`、`{{agent.name}}` 等变量，以及运行上下文中的其他值。
 
 ## 4. 会话恢复行为（Session resume behavior）
 
 对于支持会话恢复的适配器，Paperclip 会存储会话 ID。
 
 - 下一次心跳会自动复用保存的会话
 - 这样可以在多次心跳之间保持上下文连续
 - 如果上下文已经陈旧或混乱，你可以重置会话
 
 建议在以下情况下重置会话：
 
 - 你对提示策略做了较大调整
 - agent 陷入了某种“死循环”或坏状态
 - 你希望从一个干净的上下文重新开始
 
 ## 5. 日志、状态与运行历史
 
 对于每一次心跳运行，你都可以看到：
 
 - 运行状态（`queued`、`running`、`succeeded`、`failed`、`timed_out`、`cancelled`）
 - 错误文本以及 stderr/stdout 摘要
 - 来自适配器的 token 使用量/成本（若可用）
 - 完整日志（单独存储在核心运行记录之外，以适配大体量输出）
 
 在本地/开发环境中，完整日志会按配置的运行日志路径存储在磁盘上。
 
 ## 6. UI 中的实时更新
 
 Paperclip 会实时向浏览器推送运行时/活动更新。
 
 你应当能实时看到：
 
 - agent 状态
 - 心跳运行状态
 - 由 agent 工作引起的任务/活动更新
 - 仪表盘、成本与活动面板中的相关变化
 
 如果连接中断，UI 会自动重连。
 
 ## 7. 常见运行模式（Common operating patterns）
 
 ### 7.1 简单自治循环（Simple autonomous loop）
 
 1. 启用定时唤醒（例如每 300 秒）
 2. 保持“分配即唤醒”开启
 3. 使用一个聚焦且清晰的提示词模板
 4. 观察运行日志，并根据情况持续调整提示/配置
 
 ### 7.2 事件驱动循环（减少持续轮询）
 
 1. 关闭定时器或将间隔设置得足够长
 2. 保持“分配即唤醒”开启
 3. 通过按需唤醒实现手动“轻推”
 
 ### 7.3 安全优先循环（Safety-first loop）
 
 1. 设置较短的超时时间
 2. 使用保守型的提示词
 3. 监控错误，并在需要时快速取消运行
 4. 一旦发现行为“跑偏”，就重置会话
 
 ## 8. 故障排查（Troubleshooting）
 
 如果运行连续失败，可以依次检查：
 
 1. 适配器命令是否可用（`claude`/`codex` 是否已安装并完成登录）
 2. `cwd` 是否存在且可访问
 3. 查看运行错误与 stderr 摘要，然后查看完整日志
 4. 确认超时时间是否设置得太短
 5. 重置会话后重试
 6. 如 agent 持续造成错误更新，可暂时暂停该 agent
 
 典型失败原因包括：
 
 - CLI 未安装或未完成认证
 - 工作目录配置错误
 - 适配器参数/环境变量不合法
 - 提示词过于宽泛或缺少约束
 - 进程因超时被终止
 
 关于 Claude 的特别说明：
 
 - 如果在适配器环境变量或主机环境中设置了 `ANTHROPIC_API_KEY`，Claude 将使用 API key 认证而不是订阅登录。Paperclip 会在环境测试中将此作为警告展示，而不是硬错误。
 
 ## 9. 安全与风险提示（Security and risk notes）
 
 本地 CLI 适配器是在宿主机上“非沙箱化”运行的。
 
 这意味着：
 
 - 提示词中的指令非常重要
 - 配置的凭证与环境变量非常敏感
 - 工作目录的访问权限同样关键
 
 尽量从“最小权限”开始配置，除非确有必要，否则避免在广泛复用的提示词中暴露敏感信息。
 
 ## 10. 最简配置检查清单（Minimal setup checklist）
 
 1. 选择适配器（`claude_local` 或 `codex_local`）
 2. 将 `cwd` 设置为目标工作空间路径
 3. 配置启动阶段与正常运行阶段的提示词模板
 4. 配置心跳策略（定时与/或分配唤醒）
 5. 触发一次手动唤醒
 6. 确认运行成功，并记录会话 ID 与 token 使用量
 7. 观察实时更新，并迭代调优提示与配置
